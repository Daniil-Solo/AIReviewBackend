# Безопасность приложения

Актуально для любого проекта

## Критерии

### В проекте отсутствуют секреты в открытом виде

В проекте не должно быть:
- файла .env и любых его вариаций с реальными секретами
- API-ключей в коде


### Для управления секретами используются специализированные инструменты (опционально)

В больших проектах рекомендуется применять Vault, AWS Secrets Manager, Kubernetes Secrets и т.п. для хранения и ротации секретов.


### Используются инструменты статического анализа безопасности (SAST) в CI (опционально)

В CI/CD пайплайн включены инструменты типа Bandit для Python, которые автоматически ищут потенциальные уязвимости в коде.


### Для Production отображение информации о версиях отключено

Серверные фреймворки не должны выводить свою версию в ответах (например, в заголовке Server или в HTML-коде ошибок).
Это затрудняет подбор эксплойтов под конкретную версию.


### Логирование ошибок не раскрывает чувствительную информацию

В production-логах не должно быть:
- трассировок стека с внутренними путями
- данных пользователей (пароли, токены, персональные данные)
- конфиденциальных параметров запросов

Используйте настройки логирования, чтобы скрыть или замаскировать такую информацию.


### Загрузка файлов ограничена по размеру и типу

Принимаемые файлы должны проверяться на соответствие допустимым расширениям/ MIME-типам и размеру. 
Файлы желательно сохранять вне веб-доступа, переименовывать и сканировать антивирусом (если возможно).


### Сессионные куки и токены имеют корректные флаги безопасности

Все куки, особенно сессионные и аутентификационные, должны иметь флаги:
- HttpOnly — недоступны из JavaScript (защита от XSS)
- Secure — передаются только по HTTPS
- SameSite — Lax или Strict для защиты от CSRF
- Max-Age / Expires — ограниченное время жизни


### Пароли хранятся только в хэшированном виде с использованием современных алгоритмов

Пароли не должны храниться в открытом виде. 
Требуется использовать библиотеки для хэширования с солью: bcrypt, argon2-cffi или встроенные средства фреймворка (django.contrib.auth.hashers).


### Для работы с базами данных используются параметризованные запросы или ORM

Запрещается конкатенировать строки SQL с пользовательским вводом. 
Следует использовать ORM (SQLAlchemy, Django ORM) или параметризованные запросы (cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))), чтобы исключить SQL-инъекции.


### Настроены корректные CORS-политики (актуально для API-сервисов) 

CORS (Cross-Origin Resource Sharing) должен быть ограничен списком доверенных доменов, а не открыт для всех (*). 
Это предотвращает несанкционированный доступ к API с других сайтов.
