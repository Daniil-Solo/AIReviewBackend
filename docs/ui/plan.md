================================================================================
СИСТЕМА АВТОМАТИЧЕСКОЙ ПРОВЕРКИ УЧЕБНЫХ ПРОЕКТОВ (описание сгенерировано AI)
================================================================================

1. ОБЩЕЕ ОПИСАНИЕ
   Платформа позволяет создавать рабочие пространства (workspaces), внутри которых
   преподаватели и владельцы назначают задачи (tasks) с набором критериев оценки.
   Студенты загружают решения (solutions) в виде ZIP-архива или ссылки на GitHub.
   Система автоматически генерирует документацию проекта (ProjectDoc), проводит
   многоэтапное AI-ревью, запускает статические анализаторы (Ruff, Bandit),
   может проводить экзамен со студентом и выдавать финальный вердикт.
   Также предусмотрена модель биллинга (списание кредитов за ревью).

2. ТИПЫ УЧАСТНИКОВ (РОЛИ)

   2.1. Глобальные роли платформы
        - Администратор (is_admin = true) – управляет пользователями, видит всех,
          может пополнять баланс.
        - Обычный пользователь (is_admin = false) – регистрируется, создаёт
          пространства, участвует в них.

   2.2. Роли внутри рабочего пространства (workspace_members)
        - OWNER (владелец) – полный контроль над пространством: изменение,
          архивация, передача владения, управление ролями всех участников.
        - TEACHER (учитель) – может создавать/редактировать задачи, критерии,
          приглашения, просматривать все решения, но не может менять владельца
          или архивировать пространство.
        - STUDENT (студент) – может просматривать задачи пространства, загружать
          свои решения, проходить экзамен, видеть свои оценки.

   2.3. Неавторизованный пользователь (гость)
        - Только регистрация, вход, восстановление пароля (если реализовано).

3. ФУНКЦИИ ПО РОЛЯМ И СЦЕНАРИЯМ

   3.1. Аутентификация и регистрация (все роли, включая гостя)
        - Регистрация с подтверждением email (код на почту).
        - Повторная отправка кода (с ограничением частоты).
        - Вход по email/паролю, получение JWT-токена.
        - (для дизайна: экран регистрации, ввода кода, входа)

   3.2. Управление профилем (авторизованный пользователь)
        - Просмотр своего профиля (email, fullname, роль, баланс кредитов).
        - Изменение личных данных (полное имя, пароль).
        - История транзакций (если биллинг реализован).

   3.3. Рабочие пространства (workspaces)

       3.3.1. Создание пространства (OWNER при создании, любой пользователь)
              - Ввод названия, описания, настройка видимости.
              - При создании создатель становится OWNER.

       3.3.2. Просмотр списка пространств (участник)
              - Список пространств, где пользователь состоит, с ролью.
              - Возможность вступить по приглашению (slug + пароль).

       3.3.3. Управление пространством (OWNER, TEACHER – частично)
              - Редактирование названия, описания (OWNER/TEACHER).
              - Архивация (только OWNER).
              - Просмотр участников (все участники).
              - Изменение роли участника (только OWNER, нельзя менять OWNER).
              - Передача владения (только OWNER другому участнику).
              - Выход из пространства (STUDENT, TEACHER; OWNER не может выйти).
              - Удаление участника (OWNER).

       3.3.4. Приглашения в пространство (join rules)
              - Создание приглашения по slug (OWNER/TEACHER).
                * Установка роли (STUDENT или TEACHER, нельзя OWNER).
                * Опциональный пароль, срок действия, лимит использований.
              - Просмотр списка приглашений (OWNER/TEACHER).
              - Редактирование и удаление приглашения.
              - Проверка доступности slug (при создании).

   3.4. Критерии оценки (criteria)
        - Критерий имеет описание, теги, тип этапа (PROJECTDOC, CODEBASE, MANUAL),
          флаг публичности (is_public).
        - Любой авторизованный может создавать критерии (становится автором).
        - Автор может редактировать/удалять свои критерии.
        - Публичные критерии видны всем, приватные – только автору.
        - Просмотр списка критериев с фильтрацией по тегам, публичности, автору.
        - Получение доступных тегов (на основе созданных пользователем).

   3.5. Задачи (tasks)
        - Создание задачи внутри пространства (OWNER/TEACHER):
          * Название, описание, флаг is_active, флаг use_exam.
        - Привязка критериев к задаче с указанием веса (task_criteria):
          * Выбор критерия (из существующих), задание веса (float).
          * Можно изменять вес, удалять привязку.
        - Просмотр задачи:
          * Полная информация – для OWNER/TEACHER.
          * Публичная (без критических деталей) – для всех участников.
        - Редактирование, удаление задачи (OWNER/TEACHER).

   3.6. Решения (solutions)

       3.6.1. Создание решения (STUDENT, также TEACHER может тестировать)
              - Выбор задачи.
              - Выбор формата: ZIP (загрузка файла) или GITHUB (ссылка).
              - Для GitHub – проверка существования репозитория через API.
              - После создания решение получает статус CREATED.
              - Автоматически запускается пайплайн ревью (фоновые задачи).

       3.6.2. Просмотр решений
              - Студент видит только свои решения.
              - OWNER/TEACHER видят все решения задачи.
              - Детали решения включают: статус, ссылку/файл, результаты проверок,
                сгенерированный ProjectDoc, ответы на экзамен, вердикт.

       3.6.3. Отмена решения (автор)
              - Статус меняется на CANCELLED, пайплайн останавливается.

       3.6.4. Финальный вердикт преподавателя (OWNER/TEACHER)
              - Выставление оценки (grade 0-100) и текстового комментария.
              - Решение переходит в статус REVIEWED.

   3.7. Автоматическое ревью (пайплайн) – сквозной процесс

        3.7.1. Генерация ProjectDoc (автоматически при создании решения)
               - Шаг 1: первичная генерация по дереву проекта и коду.
               - Шаг 2: критика документа сторонней LLM.
               - Шаг 3: формирование правок на основе кода и критики.
               - Шаг 4: объединение исходного документа и правок.
               - Итог: project_doc сохраняется в solution_artifacts.

        3.7.2. Ревью по критериям (многоэтапное)
               - Этап 1: проверка критериев типа PROJECTDOC по ProjectDoc.
               - Этап 2: проверка критериев типа CODEBASE по коду (для низкой уверенности).
               - Этап 3: (планируется) агентная проверка критериев типа MANUAL.
               - Результат: список проверок (solution_criteria_checks) с комментариями,
                 статусом (SUFFICIENT, NEEDS_CODE, NEEDS_STUDENT, NEEDS_MANUAL, NOT_APPLICABLE)
                 и флагом is_passed.

        3.7.3. Статический анализ
               - Запуск Ruff и Bandit на коде проекта.
               - Результаты анализируются LLM, формируется обратная связь.
               - Данные сохраняются в steps (JSONB) решения.

        3.7.4. Экзамен (опционально, если задача use_exam = true)
               - Генерация вопросов (10) на основе ProjectDoc, слабых мест, замечаний.
               - Студенту открывается доступ к экзамену с ограничением по времени.
               - Вопросы сохраняются в solution_artifacts.questions.
               - Ответы студента сохраняются в exams.answers.
               - Проверка ответов (автоматическая или преподавателем).

        3.7.5. Финальный вердикт (вручную)
               - Преподаватель выставляет оценку и комментарий.
               - Статус решения становится REVIEWED.

   3.8. Биллинг (кредиты) – модель присутствует, реализация не завершена
        - Каждый пользователь имеет баланс кредитов.
        - Транзакции: WELCOME_BONUS (при регистрации), REVIEW (списание за ревью),
          ADMIN_TOP_UP (пополнение админом), DEPOSIT (вручную).
        - При создании решения может проверяться достаточно ли кредитов.
        - Интерфейс: отображение баланса, история транзакций.

   3.9. Администрирование (глобальный администратор)
        - Просмотр всех пользователей.
        - Создание администраторов.
        - Пополнение баланса любому пользователю.
        - Просмотр всех пространств и решений (возможно).

4. ПОЛНЫЙ СЦЕНАРИЙ ИСПОЛЬЗОВАНИЯ (ДЛЯ ДИЗАЙНА ПОТОКА)

   А. СТУДЕНТ
      1. Регистрация/вход.
      2. Вступление в рабочее пространство по приглашению (slug + пароль).
      3. Просмотр списка задач в пространстве.
      4. Выбор задачи, загрузка решения (ZIP или ссылка GitHub).
      5. Ожидание завершения автоматического ревью (можно видеть прогресс).
      6. Если требуется экзамен – пройти экзамен (ответить на вопросы, таймер).
      7. Просмотр финального вердикта (оценка, комментарий, детальные проверки).
      8. При необходимости – повторная сдача (новое решение).

   Б. ПРЕПОДАВАТЕЛЬ (TEACHER) / ВЛАДЕЛЕЦ (OWNER)
      1. Создание рабочего пространства, настройка.
      2. Создание приглашений для студентов.
      3. Создание критериев оценки (или выбор публичных).
      4. Создание задачи: описание, привязка критериев с весами, флаг экзамена.
      5. Просмотр списка решений студентов по задаче.
      6. Детальный просмотр решения: ProjectDoc, AI-отчёт по критериям,
         результаты линтеров, ответы на экзамен (если есть).
      7. Выставление финальной оценки и комментария.
      8. Управление пространством (участники, роли, приглашения).

   В. АДМИНИСТРАТОР ПЛАТФОРМЫ
      1. Просмотр всех пользователей, пространств.
      2. Ручное пополнение баланса.
      3. Создание других администраторов.

5. ДОПОЛНИТЕЛЬНЫЕ ВОЗМОЖНОСТИ ДЛЯ ИНТЕРФЕЙСА

   - Визуализация процесса ревью: статусная модель решения (CREATED, AI_REVIEW,
     WAITING_EXAM, EXAMINATION, HUMAN_REVIEW, REVIEWED, CANCELLED, ERROR).
   - Древовидное представление проекта (project_tree) для навигации.
   - Просмотр исходного кода с подсветкой синтаксиса (из project_content).
   - Отображение результатов проверки критериев в виде карточек с прогрессом.
   - Редактор markdown для описания задачи, комментариев преподавателя.
   - Таймер и форма ответов для экзамена.
   - Графики / метрики по решениям (средняя оценка, распределение ошибок).
   - Личный кабинет: история решений, накопленные кредиты.

6. ТРЕБОВАНИЯ К ИНТЕРФЕЙСУ НА ОСНОВЕ РОЛЕЙ

   - Для неавторизованного: главная страница с описанием, форма входа/регистрации.
   - Для студента: дашборд с активными пространствами и задачами, форма загрузки
     решения, страница ожидания ревью, страница экзамена, страница результатов.
   - Для преподавателя/владельца: управление пространствами, конструктор задач,
     конструктор критериев, список решений с фильтрацией, страница деталей решения
     с вкладками (ProjectDoc, AI-отчёт, линтеры, экзамен, вердикт).
   - Для администратора: админ-панель со списком пользователей и транзакций.
