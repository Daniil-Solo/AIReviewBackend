## Процесс ревью


### 1. Формирование ProjectDoc

ProjectDoc - это мини-документация о проекте, содержащая описание, стек, архитектуру и особенности реализации.

Его составление полезно для:
- ревью на основе критериев (некоторые критерии могут быть покрыты одним лишь содержанием ProjectDoc)
- финальной оценки со стороны преподавателя (помогает быстро разобраться, что за проект и что тут есть)

ProjectDoc формируется в 4 этапа:
1. Первичная генерация ProjectDoc на основе кодовой базы и дерева проекта с помощью LLM
2. Критика первой версии ProjectDoc сторонними LLM
3. Создание правок к замечаниям критики на основе кодовой базы с помощью LLM
4. Объединение первой версии ProjectDoc с правками для получения улучшенной версии с помощью LLM

Результатом этапа является project_doc, markdown-документация проекта.


### 2. Ревью на основе критериев

Ревью разбивается на 3 этапа:
1. Проверка выполнимости всех критериев на основе ProjectDoc с помощью LLM
2. Проверка выполнимаости критериев низкой уверенности из предыдудщего этапа на основе кодовой базы с помощью LLM
3. Проверка выполнимаости критериев низкой уверенности из предыдудщего этапа с помощью агента

Каждый этап отсеивает какие-то критерии, в которых есть высокая уверенность.
Например, критерий по архитектуре или стеку закрывается с помощью ProjectDoc.
А критерий по именованию переменных требует анализа всего кода, что происходит на втором этапе.
До третьего этапа доходят те, критерии, для которых либо требуется сложная обработка агентом с возможностью поиска в Интернете и более гибкой работой с репозиторием.

Результатом этапа является criteria_checks, представляющий json со следующей схемой:
```JSON
[
  {
    "code": "UNDNM",
    "confidence": "HIGH",
    "comment": "В работе соблюдается понятное наименование функций и классов",
    "value": 100
  }
]
```


### 3. Оценка работы статических анализаторов

Над кодом проекта запускаются сконфигурированные статические анализаторы ruff и bandit.
С помощью LLM определяется критичность замечаний и формируется обратная связь.


### 4. Экзаменация студента

С помощью LLM анализируются ProjectDoc, структура проекта, комментарии к критериям с низкой оценкой, обратная связь по статическим анализаторам
и готовится набор из 10 вопросов по тому, что не так в системе, как можно было бы ее улучшить, какие потенциальные проблемы видит стдуент в описываемой ситуации.

Основная идея в том, чтобы подискутировать со студентом об его проекте.

После подготовки вопросов студенту откроется возможность начать экзамен со свободным ответом. На экзамен есть ограничение по времени.

### 4. Финальный вердикт преподавателя

Преподаватель анализирует артефакты ревью и заполняет форму итогового вердикта

Форма вердикта:
- итоговая оценка от 0 до 100 (grade)
- комментарий от ревьювера-человека (comment)

Эндпоинты:
- `POST /solutions/:id/verdict`: Request {grade, comment} Response {id, task_id, format, link, grade, comment}
