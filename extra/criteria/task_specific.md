# Критерии специфичные для задачи

Актуально для любого проекта

## Критерии

### Функциональные требования к приложению

В проекте должны быть реализованы следующие возможности:
- Пользовательский блок: регистрация, аутентификация (JWT), управление личным кабинетом.
- ML-блок: выполнение предсказаний с использованием модели машинного обучения
- Биллинговый блок: учет баланса кредитов, списание средств по факту успешного выполнения задачи, интеграция с платежным шлюзом (для пополнения баланса)
- Аналитический блок: дашборд для отображения статистики использования сервиса и расхода кредитов


### Покрытие кода тестами > 70%

Покрытие кода тестами должно быть больше 70%
Если прямого упоминания нет, то необходимо запросить информацию у пользователя


### Реализована одна из маркетинговых механик биллинга

В проекте реализона один из следующих подходов (альтернативы допускаются):
- Система промокодов
  Промокод может давать фиксированную сумму кредитов (например, «WELCOME100» дает 100 кредитов) или процентную скидку на следующее пополнение баланса
  Следует реализовать проверку срока действия промокода и ограничение на количество активаций (одноразовые/многоразовые)
- Динамические скидки и уровни лояльности
  Реализация системы «уровней» пользователя (например: Bronze, Silver, Gold)
  Чем больше предсказаний совершил пользователь за месяц, тем выше его уровень, который дает автоматическую скидку на стоимость каждого предсказания (например, -5%, -10%, -20%) 
  Следует реализовать фоновую задачу, которая раз в месяц пересчитывает статус пользователя на основе статистики его активности
- Реферальная программа
  Пользователь получает уникальную ссылку; если по ней регистрируется новый клиент, оба получают бонусные кредиты.
  Следует разделить систему учета транзакция на «основной баланс» (купленные кредиты) и «бонусный баланс» (полученные по акциям), с правилами их приоритетного списания
- Кэшбэк
  Возврат части потраченных кредитов на бонусный счет при достижении определенного порога трат за неделю
  Следует разделить систему учета транзакция на «основной баланс» (купленные кредиты) и «бонусный баланс» (полученные по акциям), с правилами их приоритетного списания


### Маркетинговая механика биллинга является надежной

- При разделении баланса сначала необходимо списывать бонусны кредиты
- Пользователь не может активировать один и тот же промокод несколько раз
- Правила начисления скидки следует вынести в отдельные таблицы, чтобы ими можно было управлять через административную панель без пересборки проекта.


### Весь проект поднимается через docker-compose up

Для проекта должен быть написан docker-compose.yaml, которого достаточно для запуска всей системы


### В проекте присутствует краткий бизнес-план

Бизнес-план может содержать УТП и описание финансовой модели проекта


### В проекте настроен мониторинг

В системе присутствуют сервисы для сбора (Prometheus) и визуализации метрик (Grafana)
В коде настроена отправка метрик или эндпоинт для их получения /metrics
Если графики напрямую не упомянаются, то необходимо запросить информацию у пользователя


### Работа модели машинного обучения выполняется в фоне

API-сервис напрямую не должен вызывать и ждать ответа от модели, он должен создавать отложенную задачу (Celery, arq)


### Биллинговый блок корректно списывает кредиты

В коде соблюдена атомарность операция для работы с кредитами
